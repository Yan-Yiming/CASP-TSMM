TSMM 性能优化工作计划
更新时间：2026-06-18

一、定位

当前仓库已经有完整的评测框架、MKL/OpenBLAS 参考实现、Slurm 运行脚本和 Web 结果展示页面。后续工作的重点不再是“搭框架”，而是把当前初级版 opt.cpp 作为 baseline，在目标机器 Intel Xeon Platinum 9242 上做性能优化。

目标是：

1. required 四组问题在 row-major 和 col-major 下都正确。
2. 以 MKL dgemm 为 reference，记录每组 GFLOPS 和 speedup。
3. 优化 opt 算子，使 required 四组的几何平均 speedup 尽可能高。
4. 每次优化都保留独立实验记录，避免只凭单次波动判断。

二、性能优化总路线

1. 先建立目标机 baseline
   - 用 MKL + AVX-512 编译当前初级 opt.cpp。
   - 通过 Slurm 在 96 核、4 NUMA node 上运行 required row/col。
   - 保存 gflops.csv、gflops_summary.json、logs/tsmm_<job_id>.out。
   - 按问题规模、layout、speedup 排序，找出拖累 geomean 的项。

2. 再按规模做特化
   - TSMM 的四组 required 形状差异很大，一个通用 kernel 很难同时最优。
   - 后续 opt.cpp 应按 (m,n,k,layout) 分派到专门 kernel。
   - 每个 kernel 的核心目标不同：
     - 大输出矩阵：减少 C 写带宽压力，提高流式写入效率。
     - 小输出大 k：提高 k 维并行和归约效率。
     - 小 k 大 n：减少循环开销，提高 j 方向并行和向量吞吐。
     - 小矩阵：控制线程数，避免 OpenMP 调度开销超过计算收益。

3. 每轮优化都用数据驱动
   - 每次只改一个主要策略，例如 block size、线程数、是否 packing、是否 streaming store。
   - 每轮至少跑 required row/col。
   - 记录：
     - 编译器和编译参数。
     - OMP/MKL/NUMA 环境变量。
     - 每个 problem/layout 的 reference GFLOPS、opt GFLOPS、speedup。
     - required geomean speedup。
   - 如果某个策略只提升单项但明显拖累 geomean，需要回滚或做条件分派。

四、按问题规模的优化计划

1. (4000,16000,128)

特点：
 - 输出 C 有 6400 万个 double，约 512 MB。
 - FLOPs 约 16.384 GFLOP。
 - C 写入和 B 流式读取压力大，线程/NUMA 分布很关键。

row-major 优化方向：
 - j 方向连续，适合按 j-block 做 AVX-512 向量化。
 - 实验 IB/JB：
   - IB=4, JB=40
   - IB=6, JB=24
   - IB=8, JB=16/32
   - IB=10, JB=16
 - 测试 OpenMP 分配方式：
   - collapse(2) schedule(static)
   - j-group 外层并行，减少不同线程写相邻 C 行导致的带宽冲突。
 - 测试 _mm512_stream_pd：
   - C 基本只写一次，non-temporal store 可能减少 cache 污染。
   - 必须实测，因为 streaming store 对小块或未对齐尾部可能变慢。

col-major 优化方向：
 - A 的每个 i 在 k 维连续，但计算大规模 C 时会重复访问 A。
 - B 的每个 j 在 k 维连续，适合一次处理多个 j。
 - 实验 A packing：
   - 把 A 从 col-major 的 A[i*k+l] pack 成 Apack[l*m+i]。
   - 用 Apack 连续加载 8/16 个 i。
 - 实验 JB=8/12/16 列同时处理，IB=8/16。
 - C 按列连续写，适合 streaming store。

2. (8,16,16000)

特点：
 - 输出只有 128 个 double。
 - k 很大，计算量约 4.096 MFLOP。
 - 96 线程全开可能调度和归约开销过大。

优化方向：
 - 不走通用 tile。
 - 沿 k 维切分，每个线程计算 partial[8][16]。
 - 最后做小规模归约。
 - 固定线程数实验：
   - 4 threads
   - 8 threads
   - 16 threads
   - 24 threads
 - 比较两种实现：
   - OpenMP parallel num_threads(N)
   - 手动按 omp_get_thread_num 分配 k range。
 - 目标是减少线程启动、内存写 partial、最终归约的开销。

3. (32,16000,16)

特点：
 - k 很小，n 很大。
 - 输出约 512k double，约 4 MB。
 - 计算强度不高，循环开销和并行粒度很重要。

row-major 优化方向：
 - j 连续，适合宽 j-block。
 - 实验：
   - IB=2, JB=64
   - IB=4, JB=32/64
   - IB=8, JB=16/32
 - k=16 可完全展开或用编译器 unroll。
 - 线程数实验：
   - 24
   - 48
   - 96
 - 如果 96 线程变慢，按该规模固定较少线程。

col-major 优化方向：
 - A 很小，可以转置/pack 到 aligned buffer。
 - B 每列 16 个 double 连续。
 - C 每列 32 个 double 连续。
 - 实验一次处理 4/8/10 个 B 列，使用 AVX-512 同时输出 32 个 i。
 - 线程沿 j 方向并行，减少 collapse 造成的调度碎片。

4. (144,144,144)

特点：
 - 总计算量约 5.97 MFLOP。
 - 数据量小，线程过多通常会拖慢。
 - 需要控制 OpenMP 开销。

row-major 优化方向：
 - 实验 IB=8/9/12，JB=16/24/32。
 - 测试 num_threads=16/24/32/48。
 - 优先减少调度开销，避免 96 线程过度切分。

col-major 优化方向：
 - j 方向外层并行。
 - 每次处理 1/2/4 个列向量 B。
 - 每个 i-block 使用 AVX-512 对 k 做 dot product。
 - 同样测试较少线程数。

六、评估标准

每轮优化后用以下标准判断是否保留：

1. correctness 必须为 OK。
2. required geomean speedup 提升优先级最高。
3. 如果单项提升但 geomean 下降，默认不合入主 opt。
4. 如果 row-major 提升但 col-major 下降，必须用 layout 条件分派隔离。
5. 如果只在少量 runs 中偶然提升，需要重复运行确认。
6. 小矩阵不追求 96 线程满载，追求实际 wall time 最短。
